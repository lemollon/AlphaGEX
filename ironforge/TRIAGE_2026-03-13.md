## Phase 0: Databricks Data Layer Audit

### Caching Architecture

**Primary DB Client:** `ironforge/webapp/src/lib/databricks-sql.ts` (172 lines)

The webapp uses the Databricks SQL Statement Execution REST API via `fetch()` — no npm driver, no connection pool.

**Two functions for all DB access:**
- `dbQuery<T>(sql)` — SELECT queries (returns typed rows)
- `dbExecute(sql)` — INSERT/UPDATE/DELETE/MERGE (no return)

**Cache-busting mechanism (lines 34-38):**
```typescript
const cacheBust = `/* ts=${Date.now()} */`
const statement = `${sql} ${cacheBust}`
```

Every `dbQuery()` call appends a unique timestamp comment, changing the statement hash so the Databricks SQL warehouse's result cache (up to 24h by default) is bypassed.

**Critical finding: `dbExecute()` does NOT cache-bust (line 85-112).** This is correct because writes don't return cached results — but it means we rely entirely on the comment mechanism for reads.

#### Does the cache-bust actually work?

The Databricks SQL Statement Execution API caches results by **statement hash**. SQL comments (`/* ... */`) ARE part of the statement text sent to the API. The API does NOT strip them before hashing. **Evidence:** The CLAUDE.md documents this as a previously-confirmed fix with the instruction "NEVER remove the cache-busting comment."

However, this is a **fragile** mechanism:
1. If Databricks changes their hashing to normalize/strip comments, it silently breaks
2. There is no `cache: 'no-store'` on the `fetch()` call in `dbQuery()` (line 41-55), unlike the Tradier client which explicitly sets it
3. Vercel's edge/serverless runtime COULD add its own HTTP response caching layer

**Verdict:** The cache-bust works today but has zero defense-in-depth. A single layer of protection for a live trading system.

#### Application-level caching

**Found:**
- `_accountIdCache` in `tradier.ts` (line 383) — in-memory Map for sandbox account IDs. Process-scoped, resets on Vercel cold start. **Benign** — account IDs don't change.
- `_mtm_failure_counts` in `ironforge_scanner.py` (line 86) — in-memory dict for consecutive MTM failures. **Benign** — operational counter.

**NOT found:**
- No Redis, Memcached, or LRU cache
- No `memoize` patterns
- No SWR server-side caching (SWR is client-side only)
- No Map() or TTL caching of query results

### Data Freshness

**Round-trip when frontend calls status:**
```
Browser → Vercel Edge (Next.js API route) → Databricks SQL warehouse REST API → Delta Lake → response
       ~50ms                              ~200-500ms                                     ~50ms
```

Total: **~300-600ms** typical latency for a single query. The status endpoint fires **9 queries in parallel** via `Promise.all()`, so wall-clock time is dominated by the slowest query.

**Frontend polling intervals** (from `BotDashboard.tsx`):
- Status: every **15 seconds**
- Position monitor (live MTM): every **10 seconds**
- Tables/logs/history: every **30 seconds**

**When a position closes at 15:45:33, how long before the frontend shows it as closed?**

Path: Scanner `close_position()` → Databricks SQL `UPDATE` → Delta Lake commit → next status poll (up to 15s) → `dbQuery()` with cache-bust → response

**Worst case: ~15 seconds** (time until next SWR poll). The Delta Lake write is committed before `close_position()` returns, so there's no write lag.

### Query Correctness

**Critical question: Does the status column update BEFORE or AFTER the collateral counter?**

In the **scanner** (`ironforge_scanner.py` lines 1420-1457):
```
1. UPDATE positions SET status='closed'  ← position marked closed
2. SELECT SUM(collateral_required) WHERE status='open'  ← re-sum remaining
3. UPDATE paper_account SET collateral_in_use = {recalculated}  ← update counter
```

Steps 1-3 are **separate SQL statements**, not a transaction. There IS a window (between step 1 and step 3) where:
- Position is closed (status='closed')
- But paper_account.collateral_in_use still reflects the old value

**However**, the status API (`status/route.ts` lines 46-51) does NOT read `paper_account.collateral_in_use`. It recalculates:
```sql
SELECT COALESCE(SUM(collateral_required), 0) as actual_collateral
FROM {bot}_positions WHERE status = 'open' AND dte_mode = '{dte}'
```

**Verdict: The status API is immune to this race condition.** It always shows the correct collateral by recalculating from source-of-truth position data. The stale `paper_account.collateral_in_use` value is only used by the scanner for buying power checks between cycles — not displayed to the user.

## Phase 1: System Understanding (Complete Trade Lifecycle)

### 1. Position State Machine

```
                    ┌──────────┐
                    │  (none)  │
                    └────┬─────┘
                         │ save_position()
                         ▼
                    ┌──────────┐
         ┌─────────│   OPEN   │──────────┐
         │         └────┬─────┘          │
         │              │                │
    stale/holdover  profit_target    eod_cutoff
    expired_close   stop_loss        data_feed_failure
    orphan_close    manual_close     sandbox_not_filled_abort
         │              │                │
         ▼              ▼                ▼
    ┌──────────┐  ┌──────────┐    ┌──────────┐
    │ EXPIRED  │  │  CLOSED  │    │  CLOSED  │
    └──────────┘  └──────────┘    └──────────┘
```

States: `open`, `closed`, `expired`
Transitions: Only `open` → `closed` or `open` → `expired`. No intermediate states.

### 2. Collateral Lifecycle

**ALLOCATION** — `executor.py` line 318:
```python
self.db.update_paper_balance(collateral_change=total_collateral)
```
Called in `open_paper_position()` AFTER `save_position()` succeeds. Two separate operations — NOT atomic.

**RELEASE** — Two paths:

**Path A: Scanner close** (`ironforge_scanner.py` lines 1438-1457):
```python
# 1. UPDATE position status to 'closed'
# 2. SELECT SUM(collateral_required) from remaining open positions
# 3. UPDATE paper_account SET collateral_in_use = {recalculated sum}
```
Collateral is RECALCULATED from remaining open positions, not decremented. This is self-healing — even if a prior release failed, the next close recalculates correctly.

**Path B: Python executor close** (`executor.py` lines 507-510):
```python
self.db.update_paper_balance(
    realized_pnl=realized_pnl,
    collateral_change=-position.collateral_required,
)
```
Collateral is DECREMENTED. This path is used by the Python `trader.py` (local mode), not the Databricks scanner.

**Path C: Webapp force-close/eod-close** (`force-close/route.ts` lines 131-150):
```typescript
const remainingCollateral = await dbQuery(
  `SELECT COALESCE(SUM(collateral_required), 0) AS total_collateral
   FROM ${botTable(bot, 'positions')}
   WHERE status = 'open' AND dte_mode = '${dte}'`
)
await dbExecute(
  `UPDATE paper_account SET collateral_in_use = ${actualCollateral}...`
)
```
Collateral is RECALCULATED — same self-healing pattern as the scanner.

**Reconciliation** — Scanner runs collateral reconciliation EVERY scan cycle (`ironforge_scanner.py` lines 2161-2198):
```python
if abs(stored_coll - actual_coll) > 0.01:
    # Fix drift
    db_execute(f"UPDATE paper_account SET collateral_in_use = {actual_coll}...")
```

**Verdict: The collateral system has triple redundancy.** Recalculation in the close path, recalculation in the webapp endpoints, and per-cycle reconciliation in the scanner. Stuck collateral should self-heal within one scan cycle (5 minutes).

### 3. Scanner Flow (Databricks `ironforge_scanner.py`)

**One scan cycle** (`scan_bot()`, lines 2220-2380):
```
1. Load paper_account row (balance, collateral, trades)
2. Count open positions
3. If open positions → monitor_position() (check PT/SL/EOD/stale/MTM failures)
4. Determine if can open more:
   - max_trades=0 (INFERNO): always can open more
   - max_trades=1 (FLAME/SPARK): only if 0 open positions
5. If market closed → log "outside_window"
6. If can open + in entry window:
   a. get_quote("SPY"), get_quote("VIX")
   b. try_open_trade() → generates signal, sizes, executes
   c. Result: "traded:{position_id}" or "skip:{reason}"
7. If past entry cutoff → log "outside_entry_window"
8. Collateral reconciliation (every cycle)
9. Save equity snapshot
10. Write heartbeat to bot_heartbeats
11. Write log entry
```

**`sandbox_user_not_filled` generation** (`ironforge_scanner.py` line 1952):
This fires when FLAME tries to open a trade, mirrors it to the User sandbox account, and the sandbox order either:
- Returns no order_id
- Returns an order_id but the fill price readback returns None/0 within 45 seconds

The result: `return "skip:sandbox_user_not_filled"` — the entire trade is aborted. No paper position is created.

**This is NOT an error** — it's the 1:1 sync gate working as designed. If Tradier sandbox can't fill, FLAME doesn't paper-trade. This ensures paper P&L exactly matches sandbox P&L.

**But:** On March 13, EVERY FLAME scan returned this. That means the User sandbox account consistently couldn't fill orders all day. Possible causes:
1. User sandbox account has insufficient buying power (most likely)
2. Tradier sandbox was down or throttled
3. The account's option approval level doesn't support IC orders

### 4. Webapp Close Flow

**Three close endpoints:**

| Endpoint | When Available | Logic |
|----------|---------------|-------|
| `force-close` | Anytime | Single position. Gets MTM price from Tradier. Mirrors to sandbox. Recalculates collateral. |
| `eod-close` | After 2:45 PM CT only | All open positions. Same logic per position. Batched paper_account update. |
| `fix-collateral` (POST) | Anytime | Closes stale/expired/orphan positions. Reconciles paper_account from scratch. |

**Does the webapp have cascade close?** No. The webapp sends close orders to sandbox accounts via `closeIcOrderAllAccounts()`, but if the sandbox rejects the order (400 error), the paper position is still closed in the database. The sandbox failure is non-fatal — caught and logged as a warning.

**Key difference from scanner:** The webapp doesn't abort on sandbox failure. It closes the paper position regardless. The scanner (for FLAME) DOES abort if User sandbox doesn't fill on OPEN — but not on close.

### 5. Paper Trading vs Live

**When Tradier sandbox rejects an order during CLOSE:**
- Paper position IS still closed in the database (status='closed', realized_pnl set)
- Collateral IS released (recalculated from remaining open positions)
- The sandbox position may remain open at Tradier — orphaned
- This creates a sandbox/paper divergence but does NOT affect paper account accuracy

**Paper balance is NOT tied to Tradier fills for SPARK/INFERNO.** Only FLAME uses Tradier fills as source of truth. SPARK and INFERNO calculate P&L from bid/ask estimates.

Both screenshots show RECOVERY entries:

- FLAME: "Reconciled account: balance_drift($513.00), pnl_mismatch($513.00)"

These are from the scanner's per-cycle balance reconciliation. When the scanner detects that `paper_account.current_balance != starting_capital + SUM(realized_pnl)`, it logs the drift and corrects it.

**$232 drift for SPARK, $513 drift for FLAME** — this is significant. It means the paper_account balance was wrong by those amounts before the scanner fixed it. The root cause of the drift needs investigation via the diagnostic queries.

## Phase 4: Root Cause Analysis

### Root Cause 1: FLAME Blocked by Sandbox Account Health

**What happened:** FLAME couldn't trade on March 13 because the Tradier User sandbox account consistently failed to fill Iron Condor orders.

**Why it happened:** The scanner's 1:1 sync gate (`ironforge_scanner.py` lines 1938-1952) aborts the entire trade if the User sandbox doesn't fill. This is by design — FLAME paper P&L must match Tradier sandbox P&L.

**Why safeguards didn't catch it:** The log correctly shows `sandbox_user_not_filled` for every scan, but there's no escalation mechanism. The system logged the failure 50+ times with no alert, no dashboard warning, no notification. A human reviewing the FLAME dashboard would see "no trades today" but would need to read the logs to understand why.

**Systemic risk:** If the sandbox account runs out of buying power, FLAME silently stops trading for days/weeks with no proactive alert. The operator only discovers it by checking the dashboard or logs.

### Root Cause 2: Paper Account Balance Drift ($232 SPARK, $513 FLAME)

**What happened:** `paper_account.current_balance` drifted from the expected value (starting_capital + sum of realized_pnl) by $232 (SPARK) and $513 (FLAME).

**Why it happened:** Without running diagnostic queries, the most likely causes are:
1. **Double-counting:** A position's realized_pnl was added to the balance twice (once in close path, once in reconciliation)
2. **Missing decrement:** A collateral allocation happened but the corresponding release was missed (Python `executor.py` path uses decrement, not recalculation)
3. **Stale paper_account from prior bug:** A previous version of the code had a bug that left paper_account in a wrong state, and the drift accumulated over multiple days

**Why safeguards didn't catch it:** The scanner's per-cycle reconciliation (lines 2161-2198) only reconciles **collateral**, not **balance**. The balance reconciliation happens separately and was triggered at 18:43 — this suggests it runs on a different schedule (possibly once after market close).

**Systemic risk:** If balance drift occurs mid-day, the buying power check (`buying_power > $200`) could either block valid trades or allow trades that shouldn't be taken.

### Root Cause 3: EOD Close Timing

**What happened:** SPARK's position was closed at 15:45:33 CT via "eod_cutoff" — but the entry cutoff is 14:00 CT (2:00 PM) and the EOD cutoff is 14:45 CT (2:45 PM). The close happened **60 minutes after EOD cutoff**.

**Why it happened:** The scanner runs every 5 minutes, and the scanner's position monitoring checks for EOD at each cycle. The 15:45 close is consistent with the scanner's `is_after_eod_cutoff()` check (>= 14:45 CT). But waiting until 15:45 to close means the position was held for an extra hour past the cutoff.

Looking more carefully: 15:45 CT = 3:45 PM CT. But the screenshots show UTC timestamps. If the log_time is UTC, then 15:45 UTC = 9:45 AM CT, which doesn't match (the position should have been open after 14:00 CT).

Actually, looking at the log format more carefully — the timestamps in the screenshots appear to be `2026-03-13T15:45:33` which, if stored as UTC, would be 9:45 AM CT. But the monitoring entries span from T14:17 to T19:56, which in CT would be 8:17 AM to 1:56 PM. This is consistent with market hours.

**But the entry cutoff is listed as 1400 (2:00 PM CT) in the logs.** And entries at T19:19:56 say "Past entry cutoff (14:56 CT, cutoff 1400)". This confirms the timestamps are CT, not UTC.

So T15:45:33 CT = 3:45 PM CT. The EOD cutoff is 2:45 PM CT. The close happened **exactly 1 hour after the cutoff**. This could mean:
1. The scanner was checking but MTM was failing (10-failure threshold before force-close)
2. The position was already in the close process and the actual close confirmation came at 3:45 PM
3. The scanner's monitoring loop takes priority over the EOD check

**Without scanner logs showing the close attempts**, this can't be fully diagnosed.

## Phase 5: Architecture-Level Recommendations

### Design Gap 1: No Transactional Guarantees on Financial Operations

**Problem:** Position open, close, and balance update are separate SQL statements. A failure between any two creates inconsistency. The system compensates with reconciliation, but reconciliation runs on a timer (5-min cycle), not immediately.

**Root principle:** Financial operations must be atomic.

**Recommendation:** Databricks Delta Lake supports multi-statement transactions. Wrap the close+balance_update sequence in a single transaction:

```sql
-- Scanner close path should be:
BEGIN;
UPDATE positions SET status='closed', close_price=X, realized_pnl=Y WHERE position_id='Z';
UPDATE paper_account SET
  collateral_in_use = (SELECT COALESCE(SUM(collateral_required), 0) FROM positions WHERE status='open'),
  current_balance = starting_capital + (SELECT COALESCE(SUM(realized_pnl), 0) FROM positions WHERE status IN ('closed','expired')),
  ...
WHERE dte_mode = '{dte}';
COMMIT;
```

**Complexity:** M (Medium). Databricks supports transactions on Delta tables. Both scanner and webapp close paths need updating.

**If this fix itself fails:** Transaction rolls back — position stays open, balance unchanged. The next scan cycle retries. No silent corruption.

### Design Gap 2: No Alerting on Persistent Failures

**Problem:** FLAME was blocked by `sandbox_user_not_filled` for an entire trading day with no alert. The operator must manually check the dashboard or logs.

**Root principle:** Persistent failures must escalate, not just log.

**Recommendation:** Add a failure counter that triggers an alert after N consecutive identical failures:
- If the same skip reason occurs > 5 times in a row, write a WARN-level log entry with `"PERSISTENT_BLOCK: {reason} for {count} consecutive scans"`
- Add a `bot_alerts` table and a dashboard indicator that shows persistent blocks
- The status API should return `persistent_block: { reason, count, since }` if count > 5

**Complexity:** S (Small). Counter logic + one new field in status response + UI indicator.

**If this fix itself fails:** Same as today — silent failure. No worse than current state.

### Design Gap 3: Balance Reconciliation is Delayed

**Problem:** Balance drift existed for an unknown duration before the scanner's reconciliation caught it at 18:43 (after market close). During trading hours, buying power calculations used the drifted balance.

**Root principle:** Source-of-truth calculations should be computed, not stored+updated.

**Recommendation:** The status API already recalculates balance from positions (source of truth). Extend this principle:
1. **Scanner buying power check** should recalculate balance from positions on every cycle, not read `paper_account.current_balance`
2. **paper_account table** becomes a cache/convenience, not the source of truth
3. Balance formula: `starting_capital + SUM(realized_pnl FROM closed/expired positions)`

This is already how the status API works — extend it to the scanner's decision logic.

**Complexity:** S (Small). Change the scanner's balance read to a JOIN/subquery.

**If this fix itself fails:** Falls back to stored value — same as current behavior.

### Design Gap 4: `dbQuery()` Cache-Busting is Fragile

**Problem:** The only cache protection is a SQL comment timestamp. If Databricks changes their cache key algorithm to normalize comments, all reads become stale.

**Recommendation:** Add `cache: 'no-store'` to the `fetch()` options in `dbQuery()`:
```typescript
const res = await fetch(url, {
  method: 'POST',
  cache: 'no-store',  // ← ADD THIS
  headers: { ... },
  body: JSON.stringify({ ... }),
})
```

This adds a second layer of cache protection (HTTP-level) alongside the SQL comment (Databricks-level). The Tradier client already does this.

**Complexity:** S (Small). One line change.

**If this fix itself fails:** The SQL comment cache-bust still works as the primary defense.

### Design Gap 5: No Close Cascade in Webapp

**Problem:** The webapp's `force-close` and `eod-close` endpoints send a single 4-leg IC close order to Tradier sandbox. If Tradier rejects it (400 error), the sandbox position stays open. The paper position IS closed, creating divergence.

**Root principle:** Close operations should have fallback paths.

**Recommendation:** Add cascade close logic to the webapp's Tradier close path:
1. Try 4-leg IC close
2. If 400 → try 2×2-leg (put spread + call spread separately)
3. If 400 → try 4 individual leg closes
4. If all fail → close paper position anyway, log sandbox orphan

This already exists conceptually in the AlphaGEX main codebase (see CLAUDE.md "Common Mistakes" #8). Port it to IronForge's `tradier.ts` `closeIcOrderAllAccounts()`.

**Complexity:** M (Medium). Requires modifying the Tradier client's close logic.

**If this fix itself fails:** Same as today — paper position closed, sandbox position orphaned. No financial impact on paper trading, but sandbox diverges.

## Phase 6: Frontend Reliability Audit

### Data Accuracy

| Dashboard Field | Source | Computation | Verified |
|----------------|--------|-------------|----------|
| **Balance** | Status API | `starting_capital + SUM(realized_pnl)` from positions table | ✅ Live from DB |
| **Realized PnL** | Status API | `SUM(realized_pnl)` from closed/expired positions | ✅ Live from DB |
| **Unrealized PnL** | Status API | Live Tradier MTM for each open position | ✅ Live from Tradier |
| **Total PnL** | Status API | `realized + unrealized` | ✅ Server-computed |
| **Collateral** | Status API | `SUM(collateral_required)` from open positions | ✅ Live from DB |
| **Buying Power** | Status API | `balance - collateral` | ✅ Server-computed |
| **Open Positions** | Status API | `COUNT(*)` from open positions | ✅ Live from DB |
| **Total Trades** | Status API | `COUNT(*)` from closed/expired positions | ✅ Live from DB |
| **Return %** | Status API | `total_pnl / starting_capital * 100` | ✅ Server-computed |

**All values are server-computed from live DB queries.** The frontend does NOT compute balance, collateral, or buying power client-side. No stale derived values.

**The status API does return `high_water_mark` and `max_drawdown` from the `paper_account` table (stale stored values), not recalculated.** These could drift. Low severity — they're not used for trading decisions.

### Error Visibility

| Scenario | User Sees | Adequate? |
|----------|-----------|-----------|
| Status API returns 500 | Red error box: "Failed to load {bot} status" | ✅ Clear |
| Position monitor unavailable | Amber banner: "Live position monitor unavailable" | ✅ Clear |
| Force-close fails | `alert()` with error message | ⚠️ Works but not elegant |
| EOD-close rejected (before 2:45) | JSON error in console (400 response) | ❌ Not visible to user |
| Scanner offline (>15 min) | Red dot + "Scanner offline: Xm ago" | ✅ Clear |
| Sandbox fill failure | No indicator on dashboard | ❌ Must read logs |
| Balance drift detected | No indicator (reconciliation happens silently) | ❌ Invisible |

### Control Gaps

**Force-close accessibility:** The "Force Close" button is visible in every position card in the Positions tab. It requires a confirmation modal. **Accessible and visible** — no issue here.

**EOD-close accessibility:** The `eod-close` endpoint is time-gated to after 2:45 PM CT (server-side check). Before that time, it returns 400. **The user cannot force an early EOD close.** This is by design but created frustration on March 13 when the user wanted to close positions before 2:45 PM.

**Reconcile now:** The `fix-collateral` endpoint exists (GET for diagnosis, POST for fix). **But there's no button for it in the UI.** The user must know the API URL and call it directly via browser. This should be a button on the dashboard.

**Emergency close all:** `eod-close` is the closest equivalent, but it's time-gated. No "close all regardless of time" button exists. `force-close` works on individual positions only.

### Stale State Indicators

**Scanner health:** Green/yellow/red dot based on heartbeat age. ✅ Good.

**Data freshness:** No "Last updated: X seconds ago" indicator anywhere. The user has no way to know if the displayed data is from the latest poll or a stale SWR cache. ❌ Missing.

**Failed refresh:** If a SWR fetch fails, SWR keeps showing the last successful data with NO indication it's stale. ❌ Dangerous.

**"Scanning..." badge:** The header shows "Scanning..." with a pulse animation. This is based on the scanner countdown timer (computed from heartbeat timestamp), not real-time scanner status. It's a reasonable approximation but could show "Scanning..." even if the scanner is dead and just happened to have a recent heartbeat. ⚠️ Slightly misleading.

### Frontend Reliability Summary

**What's good:**
- All financial values computed server-side from DB source of truth
- Status API recalculates everything live (no stale paper_account dependency)
- Error boundaries isolate tab failures
- Force-close button visible and accessible
- Scanner health indicator with color-coded freshness

**What's missing:**
1. **No staleness indicator** — user can't tell if data is 2 seconds or 30 seconds old
2. **No sandbox failure indicator** — FLAME being blocked all day was invisible on the dashboard
3. **No "fix collateral" button** — requires API knowledge to invoke
4. **No "close all NOW" button** — eod-close is time-gated, force-close is per-position
5. **No balance drift indicator** — reconciliation happens silently
6. **SWR shows stale data on fetch failure** with no visual warning
7. **No auto-refresh after force-close** — user waits up to 15s to see the result reflected
